ps -p "$PIDS" -o pid=,ppid=,%cpu=,%mem=,stat=,wchan=,args= 명령어는 리눅스 인프라 엔지니어들이 서버 장애를 부검할 때 사용하는 가장 강력하고 정교한 메스(해부 도구)와 같습니다.

각 컬럼 뒤에 붙은 = 기호는 "헤더(제목)를 출력하지 말고 순수 데이터만 뱉어라"라는 뜻이며(스크립트에서 텍스트 파싱을 쉽게 하기 위함), 지정된 7개의 컬럼이 각각 어떤 의미를 가지고 왜 중요한지 시니어 엔지니어의 관점에서 상세히 설명해 드리겠습니다.

1. pid (Process ID)
의미: 운영체제가 프로세스를 생성할 때 부여하는 고유한 주민등록번호입니다.

트러블슈팅 활용: "어떤 놈을 쏠 것인가?"

프로세스를 추적하거나 kill -9 <PID> 명령어로 강제 종료할 때 사용되는 가장 기본적이고 핵심적인 식별자입니다.

2. 👨‍👦 ppid (Parent Process ID)
의미: 이 프로세스를 낳은(fork) 부모 프로세스의 PID입니다.

트러블슈팅 활용: "누가 좀비를 방치했는가?"

방금 전 우리가 겪었던 좀비(Zombie) 사태의 핵심 키(Key)입니다. 리눅스에서는 자식이 죽으면 부모가 시체를 수거해야 하는데, 부모가 멈추거나 문제가 생겨 수거하지 않으면 좀비가 됩니다. 이때 이 ppid를 보고 "아, 이 좀비의 부모가 1234번이구나. 1234번을 죽여야 좀비가 사라지겠네!" 하고 추적할 수 있습니다.

ppid가 1이라면 부모가 죽어서 리눅스 시스템(init/systemd)에 입양된 고아(Orphan) 프로세스임을 의미합니다.

3. ⚡ %cpu (CPU Usage Percentage)
의미: 프로세스가 사용하는 CPU의 점유율입니다.

트러블슈팅 활용: "누가 시스템을 느리게 만드는가?"

멀티코어 환경에서는 1코어를 100% 쓰면 100%, 2코어를 꽉 채워 쓰면 200%로 표기됩니다.

우리가 [CpuWorker] 폭주 테스트를 할 때, 갑자기 5%, 15%, 50%로 치솟던 바로 그 범인을 잡아내는 지표입니다. 반대로 데드락일 때는 완벽하게 0.0%를 가리키며 프로세스가 굳어있음을 증명합니다.

4. 🧠 %mem (Memory Usage Percentage)
의미: 서버의 전체 물리 메모리(RAM) 대비 이 프로세스가 차지하고 있는 물리 메모리 점유율입니다.

트러블슈팅 활용: "누가 메모리를 잡아먹어 OOM을 유발하는가?"

우리가 [MemoryWorker]의 메모리 누수를 추적할 때 유용한 지표입니다. 다만, % 단위는 서버 RAM 크기에 따라 달라지므로, 실무에서는 이 값과 함께 정확한 MB 용량을 아는 rss (Resident Set Size) 지표를 섞어 쓰는 것을 선호합니다.

5. 🚥 stat (Process State)
의미: 프로세스의 현재 운영체제 레벨 상태를 나타내는 알파벳 코드입니다.

핵심 상태 코드:

R (Running): 열심히 CPU를 쓰며 연산 중인 상태.

S (Sleeping): 무언가(네트워크, 키보드 입력, 타이머, 락 등)를 기다리며 잠들어 있는 상태. (CPU 소모 없음)

Z (Zombie): 실행을 마치고 죽었지만, 부모가 시체를 안 치워 명부에만 껍데기가 남은 상태. (아까 우리가 본 ZN의 정체!)

l (소문자 L): 프로세스 안에 멀티스레드(여러 개의 워커)가 존재한다는 뜻 (예: Sl = 잠들어 있는 멀티스레드 앱).

+: 포그라운드(화면 앞)에서 돌고 있다는 뜻.

트러블슈팅 활용: 프로세스가 정상 연산 중인지(R), 멈춰 굳어버렸는지(S), 이미 죽은 시체인지(Z)를 커널의 목소리로 확정 지어 줍니다.

6. 🔒 wchan (Wait Channel) - 가장 고급 지표
의미: 프로세스가 S (Sleeping) 상태일 때, 리눅스 커널의 어느 함수 안에서 멈춰서 자고 있는지 그 함수의 이름을 보여줍니다.

트러블슈팅 활용: "어쩌다 멈췄는가? (스모킹 건)"

do_wait: 정상입니다. 부모가 자식이 끝나길 느긋하게 기다리고 있는 안전한 상태입니다.

poll_s 또는 epoll_: 정상입니다. 네트워크 요청이나 타이머를 기다리는 중입니다.

futex: 데드락(교착상태)의 결정적 증거입니다. 프로세스가 멀티스레드 간에 자원 락(Lock)을 획득하려고 커널의 동기화 파이프(futex)에 줄을 섰는데, 앞사람이 영원히 나오지 않아 무한 대기에 빠졌음을 증명합니다.
